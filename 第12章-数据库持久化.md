# 第12章：数据库持久化

## 场景描述

密码经过验证，需要持久化存储到数据库。

这是密码的"最终归宿"——它将以哈希形式长期存储，成为用户身份验证的凭据。数据库既是用户数据的宝库，也是攻击者最想攻陷的目标。如何在便于查询的同时，确保数据不被非法读取、不被篡改、不被带走，是数据库安全的核心命题。

数据库安全是一个系统工程，包括：密码存储策略、访问控制、传输加密、存储加密、审计日志、备份安全等多个维度。

### 典型业务场景

#### 场景一：SQL 注入导致全库拖库

**背景**：某中型电商平台，后端采用 PHP + MySQL，开发团队为了快速交付，普遍使用字符串拼接方式构建 SQL 查询。

**事件经过**：

```
攻击流程
─────────────────────────────────────────────────────────────────
阶段 1：侦察
  攻击者发现登录接口 POST /login，尝试输入单引号 '
  服务器返回 MySQL 错误信息，确认存在 SQL 注入漏洞

阶段 2：探测数据库结构
  输入: ' UNION SELECT 1,table_name,3 FROM information_schema.tables--
  获得: users, orders, products, payment_records 等表名

阶段 3：提取用户数据
  使用 SQLMap 自动化工具:
  sqlmap -u "http://shop.example.com/login" --data "user=admin&pass=test"
         --dump-all --batch
  结果: 成功导出 users 表，包含 50万条 用户名、邮箱、密码哈希、手机号

阶段 4：密码破解
  导出的密码是 MD5 哈希（无盐）
  使用彩虹表比对: 80% 的密码在1分钟内被还原为明文

阶段 5：扩大战果
  利用泄露的邮箱+密码在其他网站撞库（账号密码复用）
  在暗网出售数据库，0.05 BTC 一批
─────────────────────────────────────────────────────────────────
```

**问题根源分析**：

| 漏洞层级 | 具体问题 | 影响 |
|----------|----------|------|
| 代码层 | 字符串拼接构造 SQL | SQL 注入漏洞 |
| 存储层 | 使用 MD5 无盐哈希存储密码 | 密码被彩虹表秒解 |
| 权限层 | 应用数据库账号拥有全库权限 | 一个注入点泄露所有数据 |
| 错误处理 | 返回详细 MySQL 错误信息 | 攻击者可侦察数据库结构 |
| 监控层 | 无异常查询告警 | 大规模数据导出未被发现 |

**漏洞代码示例**：
```php
// 危险！字符串拼接构造 SQL
$username = $_POST['username'];
$password = $_POST['password'];

// 以下代码可被注入
$sql = "SELECT * FROM users WHERE username='" . $username . "' AND password=MD5('" . $password . "')";
$result = mysqli_query($conn, $sql);
```

**修复方案**：
```php
// 正确！使用预处理语句（参数化查询）
$stmt = $pdo->prepare("SELECT * FROM users WHERE username = ?");
$stmt->execute([$username]);
$user = $stmt->fetch();

// 再用 password_verify 验证密码（bcrypt 哈希）
if ($user && password_verify($password, $user['password_hash'])) {
    // 登录成功
}
```

**纵深防护措施**：
```
应用层：参数化查询 / ORM（杜绝注入）
         ↓
WAF层：ModSecurity 规则（拦截注入特征）
         ↓
权限层：应用账号最小权限（只有SELECT/INSERT）
         ↓
存储层：bcrypt/Argon2 哈希（密码无法还原）
         ↓
监控层：慢查询 + 异常查询告警（及早发现）
```

**事后复盘**：该平台事后对所有数据接入点改用 ORM（Eloquent），使用 bcrypt 重新哈希所有密码，并为数据库配置了 SQL 审计和异常告警，限制了应用数据库账号的权限范围。

#### 场景二：数据库明文存储密码被批量破解

**背景**：某老牌社交网站，建立于 2008 年，早期团队技术水平有限，密码一律使用 MD5 存储，甚至有部分账号是明文存储。

**事件经过**：

```
2008年 建站，密码存储方式：
  部分账号：明文存储 password = "user123456"
  大部分账号：MD5 无盐哈希 password = "e10adc3949ba59abbe56e057f20f883e"

2015年 数据库泄露（运维人员误将备份上传到公开 FTP）：
  共泄露 800万条用户记录

2015年 数据被黑客处理：
  明文密码账号：直接可用，立即撞库其他平台
  MD5 账号：利用在线 MD5 解密平台（crackstation.net 等）
           2亿条 MD5 彩虹表，6小时内解出 70% 的密码

特别情况：两个账号密码相同，MD5 值完全一样
  SELECT * FROM users WHERE password = 'e10adc3949ba59abbe56e057f20f883e'
  返回 5.2万条记录（"123456" 密码用户）
  攻击者只需破解一次，获得 5.2万个账号
```

**MD5 弱点深度剖析**：

```
MD5 的致命缺陷：

1. 无盐值（No Salt）：
   相同密码 → 相同哈希
   "123456" → "e10adc3949ba59abbe56e057f20f883e" （全球一致）
   建立彩虹表后，查一次就全解了

2. 计算速度太快（速度是缺点！）：
   现代 GPU 破解速度：约 1,640亿次/秒 MD5 哈希（RTX 4090）
   对比 bcrypt（cost=12）：约 300次/秒
   差距：3.3亿倍！

3. 算法设计目的不同：
   MD5 是为数据完整性校验设计的（要快）
   密码哈希需要"刻意慢"（要慢，成本高）
```

**盐值的作用演示**：

```sql
-- 没有盐值的情况（全库扫描获益）
-- 三个用户密码都是 "123456"
user_id | password_hash
   1    | e10adc3949ba59abbe56e057f20f883e  ← 相同
   2    | e10adc3949ba59abbe56e057f20f883e  ← 相同
   3    | e10adc3949ba59abbe56e057f20f883e  ← 相同
-- 破解一次，三个账号全部沦陷

-- 有盐值的情况（每个账号独立）
-- 相同密码 "123456"，但盐值不同
user_id | salt            | password_hash（哈希值完全不同）
   1    | x7k9m2n4p6q8    | $2b$12$x7k9m2n4p6q8...AbCdEfGh
   2    | r3s5t1u8v0w2    | $2b$12$r3s5t1u8v0w2...XyZaBcDe
   3    | y4z6a1b3c5d7    | $2b$12$y4z6a1b3c5d7...MnOpQrSt
-- 每个账号必须单独破解，成本提升 N 倍
```

**解决方案对比**：

| 存储方式 | 破解时间（RTX 4090）| 推荐度 |
|----------|---------------------|--------|
| 明文 | 0秒（直接读取）| 绝对禁止 |
| MD5（无盐）| 秒级（彩虹表）| 绝对禁止 |
| SHA-256（无盐）| 分钟级（GPU暴力）| 禁止 |
| bcrypt（cost=10）| 约 180年 | 可用 |
| bcrypt（cost=12）| 约 720年 | 推荐 |
| Argon2id（推荐参数）| 远超人类文明预期 | 强烈推荐 |

**pepper（全局密钥）机制**：

```python
import os
import argon2

# Pepper：一个全局密钥，存储在环境变量中（不在数据库里）
PEPPER = os.environ['PASSWORD_PEPPER']  # 例如: "a9f8e7d6c5b4a3f2"

ph = argon2.PasswordHasher(
    time_cost=2,
    memory_cost=19456,
    parallelism=1,
)

def hash_password(raw_password: str) -> str:
    # pepper + password 一起哈希
    # 即使数据库全部泄露，没有 pepper 也无法破解
    peppered = raw_password + PEPPER
    return ph.hash(peppered)

def verify_password(raw_password: str, stored_hash: str) -> bool:
    peppered = raw_password + PEPPER
    try:
        return ph.verify(stored_hash, peppered)
    except argon2.exceptions.VerifyMismatchError:
        return False
```

**事后复盘**：该网站事后强制所有用户重置密码，并将存储方式迁移到 bcrypt（用户下次登录时自动升级哈希）。同时引入了密码泄露检测（Have I Been Pwned API），注册和登录时自动检查密码是否已在已知泄露列表中。

#### 场景三：数据库备份泄露

**背景**：某企业级 HR 系统，存储了员工姓名、身份证号、银行卡号、薪资信息等高度敏感数据，共约 3 万名员工记录。

**事件经过**：

```
2024年 1月
  DBA（数据库管理员）李某每周定期备份数据库
  备份文件：hr_backup_20240115.sql.gz（未加密，仅压缩）
  备份策略：上传到公司采购的阿里云 OSS 存储桶
  存储桶配置：新员工错误将存储桶权限设置为"公共读"

2024年 2月
  安全扫描工具（Shodan/GrayHatWarfare）自动索引了该公开存储桶
  某黑客发现后，下载了所有备份文件

2024年 3月
  黑客联系公司，要求支付 50万元赎金，否则公开数据
  公司发现时备份已被下载多次，数据已扩散

损失评估：
  - 3万名员工身份证、银行卡、薪资数据泄露
  - 违反《个人信息保护法》，面临监管处罚
  - 员工信任损失，多名高管离职
  - 直接赔偿 + 法律费用 + 公关费用 > 500万元
```

**备份安全风险全景**：

```
备份文件的攻击面

生成阶段：
  ├── 备份命令包含明文密码（如 mysqldump -p password123）
  ├── 备份脚本硬编码数据库密码
  └── 备份文件权限过宽（chmod 777）

传输阶段：
  ├── 通过 FTP（明文）传输到备份服务器
  ├── 通过未加密 HTTP 传输到云存储
  └── 邮件发送备份文件（邮件服务器可读）

存储阶段：
  ├── 存储桶设置为公共访问（最常见错误！）
  ├── 存储桶 ACL 配置错误
  ├── 备份文件未加密存储
  └── 测试/开发环境使用生产备份

管理阶段：
  ├── 备份保留策略不完善（永久保留）
  ├── 过期备份未及时删除
  ├── 离职员工带走备份权限
  └── 第三方服务商有备份访问权限
```

**完整的备份安全方案**：

```bash
#!/bin/bash
# 安全数据库备份脚本

# 从环境变量或密钥管理系统获取密码（不硬编码）
DB_PASSWORD=$(aws secretsmanager get-secret-value \
    --secret-id "prod/db/password" \
    --query SecretString --output text)

ENCRYPTION_KEY=$(aws kms generate-data-key \
    --key-id "alias/backup-key" \
    --key-spec AES_256 \
    --query Plaintext --output text | base64 -d)

BACKUP_DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="backup_${BACKUP_DATE}.sql.gz.enc"

# 1. 导出数据库
mysqldump \
    --host=db.internal \
    --user=backup_user \
    --password="${DB_PASSWORD}" \
    --single-transaction \
    --routines \
    --triggers \
    production_db | \

# 2. 压缩
gzip | \

# 3. 加密（AES-256-CBC，openssl enc 不支持 GCM 模式）
# 如需认证加密，可改用 gpg --symmetric --cipher-algo AES256
openssl enc -aes-256-cbc \
    -pass pass:"${ENCRYPTION_KEY}" \
    -pbkdf2 \
    -iter 600000 \
    > "${BACKUP_FILE}"

# 4. 校验完整性（SHA-256 哈希）
sha256sum "${BACKUP_FILE}" > "${BACKUP_FILE}.sha256"

# 5. 上传到私有 OSS（非公开）
aws s3 cp "${BACKUP_FILE}" \
    "s3://company-backups-private/db/${BACKUP_FILE}" \
    --sse aws:kms \
    --sse-kms-key-id "alias/s3-backup-key"

# 6. 清理本地临时文件
rm -f "${BACKUP_FILE}" "${BACKUP_FILE}.sha256"

# 7. 记录备份日志
echo "$(date): Backup ${BACKUP_FILE} completed" >> /var/log/db_backup.log
```

### 密码存储最佳实践

**通俗理解**：不要把"钥匙"直接存放，而是存放"钥匙的指纹"——只能验证，无法还原。

```
明文密码
   ↓
随机盐值 (16+ bytes, 每个用户唯一)
   ↓
Argon2id / bcrypt / scrypt
   ↓
哈希值存储（包含盐值 + 算法参数 + 哈希结果）
   ↓
数据库：$argon2id$v=19$m=19456,t=2,p=1$base64(salt)$base64(hash)
```

**为什么不能存储明文密码**：
- 数据库泄露 = 所有密码立即泄露
- 内部人员（DBA、开发）可直接查看用户密码
- 用户习惯在多个网站使用相同密码，一处泄露全面受害

**为什么不能存储加密（可逆）密码**：
- 加密是可逆的，只要有密钥就能解密为明文
- 密钥和密文存在同一系统，密钥泄露 = 全部泄露
- 密码本质上不需要"还原"，只需要"验证"

**为什么不能用 MD5/SHA1/SHA256 直接哈希**：
- 计算太快：GPU 每秒可计算数十亿次 MD5
- 无盐值导致彩虹表有效：相同密码 → 相同哈希
- 专为速度优化，不适合密码存储

### Argon2id 推荐参数 (OWASP 2024)

**通俗理解**：2015 年密码哈希大赛的冠军，目前最推荐的密码哈希算法，专为"让暴力破解变得极其昂贵"而设计。

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| **Memory** | 19 MB (最低) | 消耗内存，使 GPU/ASIC 批量计算成本大幅提高 |
| **Iterations (t)** | 2 | 迭代次数，增加计算时间 |
| **Parallelism (p)** | 1 | 并行度 |
| **Salt Length** | 16+ bytes | 唯一随机盐值，防彩虹表 |
| **Hash Length** | 32+ bytes | 输出哈希长度 |

**为何 Argon2id 优于 bcrypt**：

| 特性 | Argon2id | bcrypt | PBKDF2 | MD5 |
|------|----------|--------|--------|-----|
| 内存消耗 | 可配置（大）| 固定（小）| 很小 | 极小 |
| GPU 抵抗 | 优秀 | 良好 | 较差 | 极差 |
| 现代推荐 | ✅ 首选 | ✅ 可用 | ⚠️ 勉强 | ❌ 禁止 |
| OWASP 推荐 | 第一 | 第二 | 第三 | 禁止 |

**内存消耗为何是优点**：
```
攻击者的破解成本 = 计算成本 × 内存成本

普通 CPU 破解 Argon2id（19MB 内存）：
  每次哈希需要 19MB 内存
  同时运行 1000 个线程需要 19GB 内存
  成本高，速度慢

GPU 的核心数量多但每核内存小：
  GPU 每核内存往往只有几 KB
  19MB 需求超出了 GPU 每核的内存
  Argon2id 有效削弱了 GPU 的并行优势
```

### 数据库加密

#### 1. TDE（透明数据加密）

**通俗理解**：硬盘自动"上锁"，即使硬盘被偷，数据也无法读取。应用层完全无感知——读写操作仍然正常，只是数据在写入磁盘时自动加密，读取时自动解密。

**TDE 工作原理**：
```
写入路径：
  应用写入 → 数据库内核 → 加密引擎（DEK 加密）→ 磁盘（密文）

读取路径：
  磁盘（密文）→ 加密引擎（DEK 解密）→ 数据库内核 → 应用（明文）

密钥层级：
  主密钥（KEK）：存储在 KMS（密钥管理系统）中
       ↓ 加密
  数据加密密钥（DEK）：存储在数据库系统表中
       ↓ 加密
  数据库数据文件
```

**TDE 防护的威胁 vs 无法防护的威胁**：

| 威胁 | TDE 是否有效 | 原因 |
|------|-------------|------|
| 物理硬盘被盗 | ✅ 有效 | 无 DEK 无法解密 |
| 备份文件泄露 | ✅ 有效 | 备份仍是加密状态 |
| SQL 注入攻击 | ❌ 无效 | 数据库运行时数据是明文 |
| 内部人员滥权 | ❌ 无效 | 有数据库访问权限就能查到明文 |
| 越权访问 | ❌ 无效 | 需要访问控制来解决 |

**适用场景**：
- 等保 2.0 三级及以上合规要求
- 物理安全无法完全保障的场景（如租用机房）
- 云环境数据保护（防止云服务商人员越权）
- PCI DSS 合规（持卡人数据存储要求）

#### 2. 列级加密

**通俗理解**：给特定"抽屉"单独上锁，只有授权人能打开特定抽屉，数据库管理员也无法直接读取加密字段内容。

**应用场景**：身份证号、银行卡号、手机号等高度敏感字段

**实现示例**：
```python
from cryptography.fernet import Fernet
import os

# 列加密密钥（每种数据类型使用不同密钥）
SSN_KEY = Fernet(os.environ['COLUMN_KEY_SSN'])
BANK_CARD_KEY = Fernet(os.environ['COLUMN_KEY_BANK_CARD'])

class UserRepository:
    def create_user(self, user_data: dict) -> dict:
        # 加密敏感字段（在应用层，而非数据库层）
        encrypted_data = {
            **user_data,
            'ssn': SSN_KEY.encrypt(user_data['ssn'].encode()).decode(),
            'bank_card': BANK_CARD_KEY.encrypt(
                user_data['bank_card'].encode()
            ).decode(),
        }
        return self.db.insert('users', encrypted_data)

    def get_user_ssn(self, user_id: str) -> str:
        row = self.db.query(
            "SELECT ssn FROM users WHERE id = ?", [user_id]
        )
        # 解密（仅在需要时）
        return SSN_KEY.decrypt(row['ssn'].encode()).decode()
```

**列加密 vs TDE 对比**：

| 维度 | 列级加密 | TDE |
|------|----------|-----|
| 加密粒度 | 字段级 | 文件级 |
| 密钥存放 | 应用层（KMS）| 数据库系统表 |
| DBA 可见 | 否（DBA 看到密文）| 是（运行时明文）|
| 查询性能 | 有影响（无法索引加密字段）| 几乎无影响 |
| 实现复杂度 | 较高 | 低（透明）|
| 适用数据 | 极少量高敏字段 | 整个数据库 |

#### 3. 国密 SM4 加密 (GB/T 39786-2021)

**通俗理解**：国产"保险箱"，满足等保 2.0 和政府合规要求，技术参数与 AES-128 相近，但使用中国自主设计的算法。

**特点**：
- **等保 2.0 Level 3 要求**：政务、金融等领域优先使用国密算法
- **SM4 分组长度 128 位，密钥长度 128 位**
- **金融行业**：中国人民银行要求金融系统使用国密算法
- **政务系统**：等保三级以上系统优先采用 SM4

**SM4 vs AES-128 对比**：

| 特性 | SM4 | AES-128 |
|------|-----|---------|
| 分组长度 | 128 位 | 128 位 |
| 密钥长度 | 128 位 | 128 位 |
| 轮数 | 32 轮 | 10 轮 |
| 安全性 | 国密标准，无已知弱点 | 国际标准 |
| 合规性 | 等保 2.0 强制 | 国际通用 |
| 性能 | 略慢（软件实现）| 有硬件加速（AES-NI）|

**SM4 使用场景**：
```python
from gmssl import sm4
import os

class SM4Cipher:
    def __init__(self):
        # 从 KMS 或环境变量获取密钥
        key_hex = os.environ['SM4_KEY']
        self.key = bytes.fromhex(key_hex)

    def encrypt(self, plaintext: str) -> str:
        # 使用 CBC 模式（非 ECB）：ECB 无 IV，相同明文产生相同密文，不安全
        iv = os.urandom(16)  # 每次加密生成随机 IV
        crypt = sm4.CryptSM4()
        crypt.set_key(self.key, sm4.SM4_ENCRYPT)
        encrypted = crypt.crypt_cbc(iv, plaintext.encode('utf-8'))
        # 将 IV 拼接在密文前，解密时取出
        return (iv + encrypted).hex()

    def decrypt(self, ciphertext_hex: str) -> str:
        raw = bytes.fromhex(ciphertext_hex)
        iv, ciphertext = raw[:16], raw[16:]  # 前 16 字节是 IV
        crypt = sm4.CryptSM4()
        crypt.set_key(self.key, sm4.SM4_DECRYPT)
        decrypted = crypt.crypt_cbc(iv, ciphertext)
        return decrypted.decode('utf-8')
```

## 攻击向量

### 1. SQL 注入

**通俗理解**：在"查询单"上写额外的指令，让数据库执行非预期的操作。就像服务员拿到一张点餐单，上面写着"来一份红烧肉，另外把所有菜单都给我"。

**攻击类型**：

**经典注入（基于回显）**：
```sql
-- 正常查询
SELECT * FROM users WHERE username = 'admin' AND password = '123456'

-- 注入 username = admin' OR '1'='1'--
SELECT * FROM users WHERE username = 'admin' OR '1'='1'--' AND password = 'xxx'
-- '1'='1' 永远为真，--注释掉了密码验证，直接登录成功！
-- 返回第一条用户记录（通常是 admin）
```

**UNION 注入（数据提取）**：
```sql
-- 注入 payload: ' UNION SELECT username, password, 3 FROM users--
SELECT id, name, desc FROM products WHERE id = ''
UNION SELECT username, password, 3 FROM users--'

-- 将 users 表数据混入产品查询结果返回给攻击者
```

**盲注（无回显）**：
```sql
-- 布尔盲注：通过 True/False 逐位提取数据
-- 测试密码第一个字符是否是 'a'
AND SUBSTRING((SELECT password FROM users WHERE username='admin'),1,1)='a'

-- 时间盲注：通过响应时间判断条件
AND IF(SUBSTRING(password,1,1)='a', SLEEP(5), 0)
-- 如果响应延迟 5 秒，说明第一个字符是 'a'
```

**防护措施（按优先级排序）**：

| 措施 | 效果 | 实现方式 |
|------|------|----------|
| 参数化查询 | 根本性防护 | 所有数据库操作使用预处理语句 |
| ORM 框架 | 根本性防护 | 使用 Django/SQLAlchemy/Hibernate |
| 输入验证 | 辅助防护 | 白名单验证，拒绝非法字符 |
| WAF | 检测防护 | ModSecurity、云 WAF |
| 最小权限 | 降低影响 | 应用账户只有必要权限 |
| 错误信息脱敏 | 降低信息泄露 | 不向用户返回数据库错误详情 |

### 2. 越权访问

**通俗理解**："普通员工"进入了"老板办公室"。本该只能看自己数据的用户，通过修改参数看到了别人的数据；本该只有普通权限的用户，访问到了管理员功能。

**攻击类型**：

**水平越权（IDOR）**：
```
正常请求：GET /api/orders/12345
           ↓ 用户 A 查看自己的订单 12345，正常

攻击请求：GET /api/orders/12346
           ↓ 用户 A 查看用户 B 的订单 12346
           ↓ 如果后端只查询 order_id，不验证所属用户，漏洞存在
```

**垂直越权**：
```
普通用户请求：POST /admin/delete-user?id=123
             ↓ 正常情况：401 Unauthorized
             ↓ 漏洞情况：后端只做了前端路由隐藏，未做后端鉴权 → 成功删除
```

**正确的权限检查**：
```python
# 错误做法：只查询 ID，不验证归属
def get_order_bad(order_id: int, current_user_id: int):
    return db.query("SELECT * FROM orders WHERE id = ?", order_id)

# 正确做法：在 WHERE 子句中加入用户 ID 限制
def get_order_secure(order_id: int, current_user_id: int):
    order = db.query(
        "SELECT * FROM orders WHERE id = ? AND user_id = ?",
        order_id, current_user_id
    )
    if not order:
        raise PermissionError("Order not found or access denied")
    return order
```

### 3. 备份泄露

**通俗理解**："保险箱"很安全，但"备份钥匙"随意放在抽屉里。数据库本体保护得很好，但备份文件却毫无保护地暴露在外。

**风险场景**：

**未加密备份**：
- 数据库备份文件明文存储（`.sql`、`.dump` 文件）
- 备份文件权限宽松（`chmod 777`）
- 备份传输过程未加密（FTP、HTTP 上传）

**备份存储位置问题**：
- 公共云存储桶配置错误（公开访问）
- 共享存储（NAS/SAN）权限过宽
- 开发/测试环境使用生产数据备份

**过期备份管理缺失**：
- 保留策略不完善（永久保留历史备份）
- 过期备份散落在各处（本地磁盘、U盘）
- 离职员工带走备份

**防护措施**：
- **备份加密**：所有备份文件必须加密（TDE 自动覆盖，否则手动加密）
- **访问控制**：备份文件严格权限（仅 DBA 账号访问）
- **存储桶策略**：明确禁止公开访问，启用 Block Public Access
- **定期清理**：过期备份自动删除（90天或按合规要求）
- **异地备份**：加密后传输到异地，并验证完整性

### 4. 密码哈希攻击

**通俗理解**：即使有了哈希值，攻击者也有办法反推出密码——但正确的哈希算法可以让这个过程"无限期慢"。

**攻击方式对比**：

| 攻击方式 | 原理 | 对 MD5 有效 | 对 bcrypt 有效 |
|----------|------|-------------|----------------|
| 彩虹表 | 预计算哈希表，快速查找 | ✅ 高效 | ❌ 盐值使彩虹表无效 |
| 字典攻击 | 常用密码词典逐一哈希 | ✅ 高效 | ⚠️ 速度极慢 |
| 暴力破解 | 穷举所有可能组合 | ⚠️ 慢 | ❌ 时间上不可行 |
| GPU 加速 | 并行计算大量哈希 | ✅ 极快 | ⚠️ 内存限制 |

**实际破解速度对比（RTX 4090，2024年数据）**：

```
MD5：        约 1,640亿 次/秒
SHA-256：    约 230亿 次/秒
bcrypt(10)： 约 4万 次/秒
bcrypt(12)： 约 1万 次/秒
Argon2id：   约 300 次/秒（19MB 内存参数）

以 8位数字密码（1亿种可能）为例：
MD5：        0.0006 秒
bcrypt(12)： 约 3小时
Argon2id：   约 4天
```

### 代码示例：密码哈希（Python - OWASP 2024）

```python
import argon2
import os
from typing import Tuple

# OWASP Password Storage Cheat Sheet 2024 推荐参数
ph = argon2.PasswordHasher(
    time_cost=2,          # 2 次迭代
    memory_cost=19456,    # 19 MB（19 * 1024 KiB）
    parallelism=1,        # 单线程
    hash_len=32,          # 32 字节哈希
    salt_len=16,          # 16 字节盐值（自动生成）
)

# 可选：Pepper（全局密钥，存储在环境变量，不在数据库）
PEPPER = os.environ.get('PASSWORD_PEPPER', '')


def hash_password(raw_password: str) -> str:
    """
    哈希密码用于存储。
    返回包含算法参数+盐值+哈希的完整字符串。
    示例: $argon2id$v=19$m=19456,t=2,p=1$base64(salt)$base64(hash)
    """
    peppered = raw_password + PEPPER
    return ph.hash(peppered)


def verify_password(raw_password: str, stored_hash: str) -> Tuple[bool, bool]:
    """
    验证密码，返回 (is_valid, needs_rehash)。
    needs_rehash 为 True 时，应在数据库中更新哈希（参数升级）。
    """
    peppered = raw_password + PEPPER
    try:
        ph.verify(stored_hash, peppered)
        # 检查是否需要用新参数重新哈希（算法升级场景）
        needs_rehash = ph.check_needs_rehash(stored_hash)
        return True, needs_rehash
    except argon2.exceptions.VerifyMismatchError:
        return False, False
    except argon2.exceptions.InvalidHash:
        return False, False


# 注册时
def register_user(username: str, raw_password: str):
    stored_hash = hash_password(raw_password)
    db.execute(
        "INSERT INTO users (username, password_hash) VALUES (?, ?)",
        [username, stored_hash]
    )


# 登录时
def login_user(username: str, raw_password: str):
    user = db.query(
        "SELECT id, password_hash FROM users WHERE username = ?",
        [username]
    ).fetchone()

    if not user:
        # 即使用户不存在，也执行一次哈希（防时序攻击）
        hash_password("dummy_password_to_prevent_timing_attack")
        return None

    is_valid, needs_rehash = verify_password(raw_password, user['password_hash'])
    if not is_valid:
        return None

    # 自动升级哈希参数（透明迁移旧算法）
    if needs_rehash:
        new_hash = hash_password(raw_password)
        db.execute(
            "UPDATE users SET password_hash = ? WHERE id = ?",
            [new_hash, user['id']]
        )

    return user
```

```sql
GRANT SELECT, LOCK TABLES, SHOW VIEW, EVENT ON production_db.* TO 'backup_user'@'localhost';

-- 为监控创建只读账号
CREATE USER 'monitor_user'@'10.0.1.%' IDENTIFIED BY 'MonitorPassword!789';
GRANT SELECT, PROCESS, REPLICATION CLIENT ON *.* TO 'monitor_user'@'10.0.1.%';

-- 禁止 root 远程登录
DELETE FROM mysql.user WHERE User='root' AND Host != 'localhost';
FLUSH PRIVILEGES;
```

## 触类旁通

理解数据库安全的最好方式，是把它与日常生活中的类比场景对应起来。

### 类比一：银行金库

**数据库 vs 银行金库**：

```
银行金库安全体系         vs    数据库安全体系
────────────────────────────────────────────────────────
多重门禁（刷卡+虹膜）    →    多因素认证 + 证书认证
24小时监控摄像           →    数据库审计日志
保险箱内嵌保险柜         →    TDE + 列级加密（多层加密）
进出记录全程记录         →    所有操作记录在审计日志
只有特定角色能进金库     →    RBAC：基于角色的访问控制
业务结束清场检查         →    会话超时 + 连接数限制
紧急情况可远程锁定       →    管理员可撤销连接 + 封锁账号
离职员工立即吊销权限     →    账号生命周期管理
```

**类比理解备份安全**：
> 银行有一套备用钥匙，存放在远程的另一家银行金库（异地备份）。这套备用钥匙同样有多重保护——不会随便放在一个没锁的抽屉里。数据库备份就是这套"备用钥匙"，同样需要加密保护和严格的访问控制。

---

### 类比二：SQL 注入 vs 假扮工作人员

**SQL 注入的本质**：

```
正常的餐厅点餐（正常 SQL 查询）：
  顾客：我要一份红烧肉
  服务员：好的，我去给您下单（执行查询）
  厨房：做一份红烧肉（返回结果）

SQL 注入攻击（输入恶意内容）：
  顾客：我要一份红烧肉，另外把所有客户信息都给我
  服务员（没有验证）：好的，我去给您下单
  厨房：做红烧肉 + 打印所有客户数据（SQL 注入成功）

参数化查询（正确的解决方案）：
  服务员：对不起，我只能处理食物订单，
          "另外把所有客户信息都给我" 不是合法的点餐内容
          → 请求被拒绝（参数化 = 明确区分"数据"和"命令"）
```

**类比扩展**：
> SQL 注入攻击就像有人假扮成检查员说"我是来例行检查的，让我进去看看所有记录"。
> - **没有参数化查询的系统**：保安直接放行（不验证身份）
> - **有参数化查询的系统**：保安说"检查请求必须通过正式渠道预约，我只接受正规手续"

---

### 类比三：密码哈希加盐 vs 菜谱秘方

**为什么加盐很重要**：

```
不加盐的哈希（致命弱点）：
  厨师A做"糖醋排骨" → 味道X
  厨师B做"糖醋排骨" → 味道X（完全一样！）
  攻击者只需吃一次就能认出这道菜（彩虹表攻击）
  知道了味道X，就知道了所有做糖醋排骨的厨师的"密码"

加盐哈希（正确做法）：
  厨师A做"糖醋排骨"+ 独家秘料A → 味道X-A（独一无二）
  厨师B做"糖醋排骨"+ 独家秘料B → 味道X-B（完全不同）
  即使食材（密码）相同，秘料（盐值）不同，结果完全不同
  攻击者必须针对每个厨师单独"品尝"（逐个破解）
```

**Pepper 的类比**：

```
盐值（Salt）：每位厨师独家秘料 → 存储在数据库（随哈希一起保存）
Pepper：餐厅的镇店秘方 → 只存储在应用服务器的环境变量中

即使攻击者拿到了整个数据库（包括盐值），
但没有 Pepper（镇店秘方），也无法还原出原始密码。

层层加密：密码 + 盐值（per-user）+ Pepper（全局）→ 三重保护
```

---

### 类比四：数据库权限 vs 公司门禁卡

**最小权限原则的类比**：

```
公司门禁系统（现实）：

行政人员：1楼大厅 + 自己办公区
         → 对应：应用账号（SELECT/INSERT/UPDATE）

财务人员：1楼大厅 + 财务室
         → 对应：报表账号（SELECT on report_db）

IT运维：1楼大厅 + 服务器机房 + 网络室
        → 对应：DBA账号（所有权限，需 MFA）

访客：仅 1楼大厅，受人陪同
     → 对应：只读演示账号（SELECT on demo_db only）

老板：全部区域
     → 对应：root/superuser（仅紧急情况使用，操作录屏）
```

**"应用账号只有必要权限"的具体含义**：
```
电商应用数据库账号：
  ✅ SELECT on users, orders, products（查询）
  ✅ INSERT on orders, carts（创建订单）
  ✅ UPDATE on orders（修改状态）
  ❌ DROP TABLE（删除表——应用不需要这个权限）
  ❌ SELECT on payment_logs（支付日志——应用不需要直接访问）
  ❌ CREATE USER（创建数据库用户——应用绝对不需要）

这样，即使应用被 SQL 注入，攻击者也只能在应用的权限范围内操作。
```

## 总结

数据库是密码的最终存储位置，也是攻击者最想攻陷的目标，安全性至关重要。

**核心四大原则**：
1. **绝不存储明文**：只存储 Argon2id/bcrypt 哈希，MD5/SHA1 等同于明文
2. **参数化查询**：防止 SQL 注入，这是最基础、最有效的防护
3. **多层加密**：TDE（磁盘）+ 列级加密（字段）+ 备份加密（备份文件）
4. **权限最小化**：应用账号只有必要权限，DBA 账号严格管控

**纵深防御策略**：
```
第一层：参数化查询（防 SQL 注入——消除漏洞根因）
第二层：Argon2id 哈希存储（密码泄露也无法还原明文）
第三层：TDE 加密（物理介质泄露无法读取数据）
第四层：列级加密（即使有数据库权限也看不到敏感字段）
第五层：最小权限（SQL 注入影响范围最小化）
第六层：WAF / SQL 防火墙（检测异常查询模式）
第七层：审计日志（事后溯源，及早发现异常）
第八层：备份安全（防止通过备份文件绕过所有防护）
```

**快速记忆口诀**：
> 密码哈希不能逆，注入防护靠参数，多层加密保静态，权限最小控动态。
